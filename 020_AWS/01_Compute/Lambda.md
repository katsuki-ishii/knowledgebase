# Lambda

## Lambda 以前の世界

Lambda が登場した2014年以前、コードを動かすには**必ずサーバーが必要**だった。

2006年に EC2 が登場し「物理サーバーを買う代わりに仮想サーバーを借りる」世界になったが、本質的には**サーバー管理の責任がユーザーにある**ことは変わらなかった。OS 選択、ランタイムインストール、セキュリティパッチ、スケーリング設計、障害対策——開発者がやりたいのはビジネスロジックを書くことなのに、インフラの面倒を見る時間が大量に取られていた。

---

## Lambda の思想：「サーバーという概念を消す」

Lambda の核心的な問いはこうだ：

> **「開発者が関数（= ロジック）だけ書いて、あとは全部クラウドに任せられないか？」**

これを実現するために、Lambda は3つの根本的な設計判断をした。

### ① イベント駆動

「常に待ち受ける」のではなく「何かが起きたら動く」。HTTP リクエスト、S3 へのファイルアップロード、DynamoDB の変更、SQS のメッセージ——トリガーがあって初めて実行される。サーバーが「常駐」する必要がなくなった。

### ② 実行単位の極小化

EC2 は「マシン」を借りる。Lambda は「1回の関数実行」を借りる。粒度がまるで違う。これにより「使った分だけ課金」が可能になった。リクエストがゼロなら課金もゼロ。

### ③ スケーリングの完全自動化

1リクエストなら1つの実行環境、1000リクエストなら1000の実行環境。開発者がキャパシティプランニングをする必要がない。

---

## 責任境界の移動 — Lambda が本当に解決したもの

技術的な利便性の奥にある、もっと本質的な変化は**責任境界の移動**だ。

|                | EC2          | Lambda             |
|----------------|-------------|---------------------|
| OS管理          | ユーザー     | AWS                 |
| パッチ適用       | ユーザー     | AWS                 |
| スケーリング     | ユーザー     | AWS                 |
| 可用性設計       | ユーザー     | AWS                 |
| 課金単位        | 時間（常時）  | 実行回数 + 実行時間   |
| 障害復旧        | ユーザー     | AWS                 |

ユーザーの責任は「関数コード + IAM 設定 + アーキテクチャ設計」に絞られる。AWS はこれを**共有責任モデル**と呼んでおり、Lambda はこのモデルでユーザー側の責任を最小化した形態。

---

## 実行環境の仕組み

### microVM（Firecracker）

Lambda 関数を呼び出すと、AWS は裏側で **Firecracker** という自社開発の microVM 技術を使って軽量な Linux 環境を起動する。

| 技術               | 特徴                              | 起動速度       |
|--------------------|----------------------------------|---------------|
| EC2（従来のVM）     | 完全な仮想マシン                    | 数十秒〜分     |
| Docker コンテナ     | カーネル共有、隔離が弱い             | 秒単位         |
| Firecracker microVM | 専用カーネル + 最小構成、完全隔離    | 約125ms       |

Docker はカーネルをホストと共有するので隔離が甘い。Firecracker は専用のミニ仮想マシンを起動するので、他のテナントとカーネルレベルで完全に分離される。Lambda だけでなく Fargate の裏側もこの技術で動いている。

### ディレクトリ構造

実行環境は Linux なので、標準的な Linux のディレクトリ命名規則に従っている。

```
/                  ← ルート（全ての起点）
├── var/task/      ← 関数コードが展開される（読み取り専用）
├── var/runtime/   ← Lambda ランタイム本体（Python, Node.js など）
├── opt/           ← 追加パッケージ（レイヤー等）が展開される（読み取り専用）
├── tmp/           ← 一時ファイル領域（唯一の書き込み可能領域、512MB〜10GB）
└── ...
```

- `/opt` は Linux で元々「**optional（追加パッケージ）**」を置く場所。レイヤーの置き場として意味的にも正しい
- `/var` は「**variable（可変データ）**」。Lambda はここに関数コードを展開している
- `/tmp` は「**temporary**」。一時ファイル用で、唯一の書き込み可能領域

### コードの保存と展開

デプロイした関数コードは zip ファイルとして **AWS が内部的に管理する S3 バケット**に保存される。これはユーザーの S3 コンソールには表示されない、アカウントごとにプライベートな内部ストレージ。各アカウントにはリージョンあたり 75GB のストレージが割り当てられている。

Lambda 起動時に AWS のインフラがこの内部 S3 から zip をダウンロードし、microVM 内の `/var/task` に解凍する。

---

## 同時実行モデル

Lambda は **1実行環境 = 1リクエスト** という設計。同時に10リクエストが来たら、10個の microVM が並列で起動する。同時実行数 = 起動中の microVM 数。

- デフォルトの同時実行上限はリージョンあたり **1,000**
- 各 microVM が独立してリソースを確保するため、同時実行数が増えると DB コネクション数も線形に増える

---

## ネットワーク — VPC 統合

Lambda はデフォルトでは AWS のパブリックネットワーク上で動き、DynamoDB や S3 にはパブリックエンドポイント経由でアクセスする。

VPC 内のリソース（RDS, ElastiCache など）にアクセスする場合は Lambda を VPC に配置する必要がある。以前は起動のたびに ENI（仮想ネットワークカード）を作成していたため、コールドスタートが10秒以上遅くなることがあった。2019年に **Hyperplane** 技術の導入で ENI を事前プールする仕組みに変わり、劇的に改善された。

---

## 実行ロールと認証情報

Lambda 関数に IAM ロールをアタッチすると、実行環境内に一時的な認証情報（STS トークン）が**環境変数として注入**される。

```
AWS_ACCESS_KEY_ID=ASIA...
AWS_SECRET_ACCESS_KEY=...
AWS_SESSION_TOKEN=...
```

boto3 や AWS SDK はデフォルトでこれらの環境変数を読むため、関数コードで明示的にキーを渡す必要がない。EC2 のインスタンスメタデータサービス（169.254.169.254）と似た発想だが、Lambda は環境変数方式でよりシンプルかつ SSRF 攻撃のリスクも低い。

---

## Lambda の思想から生まれた制約

Lambda の制約は恣意的なものではなく、全て「サーバーを消す」という思想のトレードオフ。

| 制約                     | 思想的な理由                                              |
|--------------------------|----------------------------------------------------------|
| 実行時間上限（最大15分）    | 長時間動くなら「関数」ではなく「プロセス」。EC2/ECS の領域   |
| ステートレス              | 使い捨ての実行環境で自動スケールするための前提条件            |
| デプロイサイズ制限         | microVM の高速起動に小さいコードが有利                      |
| コールドスタート           | 「常駐しない」の代償                                      |

## 今の Lambda が抱える課題

### コールドスタート問題

Firecracker で速くなり、Provisioned Concurrency で緩和でき、SnapStart（Java）で改善された。しかし「常駐サーバーに比べてレイテンシが不安定」という本質は残っている。リアルタイム性が求められるワークロード（音声認識、チャットなど）では設計上のポイントになる。

### ベンダーロックイン

Lambda 関数自体は移植できても、周辺のイベントソース統合（API Gateway, SQS, DynamoDB Streams）が AWS 固有。関数の書き直しより**グルー部分の移植が圧倒的に大変**。

### デバッグ・観測性の難しさ

実行環境が使い捨てなので SSH して調査ができない。CloudWatch Logs、X-Ray、PowerTools for Lambda などの可観測性ツールに依存せざるを得ず、「何が起きたのか」を追うコストが高い。

### コスト予測の難しさ

「使った分だけ」は安く聞こえるが、リクエスト量が予測しにくいワークロードでは月末の請求が読めない。EC2 のリザーブドインスタンスのような「確実に安くする」手段が限られている。

### 「関数」の粒度問題

1つの API に1つの Lambda を作るのか、複数エンドポイントを1つにまとめるのか。細かく分ければ管理が大変、まとめればモノリスに逆戻り。アプリケーションの規模が大きくなると、ここの設計判断が効いてくる。

---

## まとめ

Lambda の面白さは、技術的な仕組みだけでなく「インフラの責任をどこまでクラウドに委譲できるか」という思想実験でもある点にある。その思想を理解した上で制約を受け入れるか、制約が合わないなら ECS や EC2 に逃がすか——この判断ができることが、サーバーレスを「使いこなしている」状態だと言える。
