# [[Terraformの歴史]]

---

## 1. なぜ[[Terraformの歴史|Terraform]]が生まれたか

### 問題の本質：「意図の揮発」

コンソールのポチポチ操作は「操作の履歴」であって「意図の記録」ではない。
なぜその構成にしたのか、何を実現しようとしたのかが消える。
[[Terraformの歴史|Terraform]]はこの**意図の揮発**という問題を解くために生まれた。

### 時系列

| 年 | 出来事 |
|---|---|
| 2011 | AWS CloudFormation登場 |
| 2014 | [[Terraformの歴史|Terraform]]登場（HashiCorp） |

CloudFormationは「AWSをコードで管理する」が目的。  
[[Terraformの歴史|Terraform]]は「複数のクラウド・SaaSを統一の思想で管理する」という、**一段上の抽象化**として設計された。

## 2. 思想的コア

### [[基礎 Part 8 宣言・代入・再代入|宣言]]的（Declarative）vs 手続き的（Imperative）

| 手続き的                 | [[基礎 Part 8 宣言・代入・再代入]]的 |
| -------------------- | --------------------- |
| 「まずVPCを作る、次にサブネットを…」 | 「この状態が正しい」            |
| Howを記述する             | Whatを記述する             |
| アセンブリ的               | SQL的                  |

[[Terraformの歴史|Terraform]]は**Whatだけ書けばいい**。そこへの経路はツールが計算する。

### Desired State（望ましい状態）

現実のインフラ状態とコードに書かれた状態を比較して、**差分だけを適用する**。

これは制御理論のフィードバック制御と同じ構造：
- エアコンが「28度」に向けて差分を埋め続けるのと同じ
- Kubernetesも同じ思想を持つ
- 共通する概念：**[[冪等性]]（[[冪等性|idempotency]]）** ——何度実行しても同じ結果になる

### [[冪等性]]を支えるコスト

優れた思想にはそれを支えるコストがある。  
[[Terraformの歴史|Terraform]]は`terraform.tfstate`で現実を追跡し続けることで[[冪等性]]を保証している。  
stateが壊れると[[冪等性]]が崩れ、Desired Stateへの収束が保証できなくなる。

## 3. HCLという言語選択

### なぜ[[設定ファイル言語|YAML]]でもJSONでもないか

| フォーマット | 問題 |
|---|---|
| JSON | コメントが書けない、引用符・カンマが多い |
| [[設定ファイル言語]] | 人間が読み書きしやすいが、スペース/タブ混在で壊れやすい |
| [[設定ファイル言語]] + CloudFormation拡張 | `!Ref` `!Sub`などを乗せると構文が歪む |

**本質的な問題**：[[設定ファイル言語|設定ファイル]]形式の上にプログラミング的表現（[[AI激動期における「定数」思考|変数]]・モジュール・ループ）を乗せると、どのフォーマットでも歪む。

HCLはその表現力を**最初から込みで設計**した言語。[[ITと不完全さに惹かれる理由|完全]]なプログラミング言語ではなく、「ちょうどいい表現力」に絞っている。

## 4. SAMとの比較

| | SAM/CloudFormation | [[Terraformの歴史]] |
|---|---|---|
| 対象 | AWSのみ | マルチクラウド |
| 強み | AWSリソースへの深い統合 | 汎用性・可読性・エコシステム |
| stateの管理 | AWSが自動管理 | 自前でstate管理が必要 |
| 依存解決 | CloudFormationが処理 | DAGで自動解決（順序が微妙に違う） |

SAM経験者の落とし穴：**「CloudFormationと同じノリで使おうとする」**こと。  
特に[[IAM ロールの PassRole と AssumeRole をもう二度と忘れないために絵を描いてみた  DevelopersIO|IAMロール]]とリソースの紐付けタイミングで詰まりやすい。

## 5. HashiCorpという会社

### プロダクト群

| ツール | 何をするか |
|---|---|
| **[[Terraformの歴史]]** | インフラのプロビジョニング |
| **Vault** | シークレット・認証情報の管理 |
| **Consul** | サービスディスカバリ・メッシュ |
| **Nomad** | ワークロードのスケジューリング |
| **Packer** | マシンイメージの構築 |

共通思想：**「インフラの複雑性を人間がコントロール可能な形に変換する」**

### 創業者

Mitchell Hashimoto（橋本）は日系アメリカ人。HashiCorpの社名は彼の名字「橋（Hashi）」から。  
インフラの**橋渡し**ツールを作る会社——偶然だが語呂がいい。
