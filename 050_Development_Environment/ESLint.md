# [[ESLint]] 基礎まとめ

## 1. [[ESLint]]とは

[[ESLint]]は JavaScript / TypeScript のコードを**実行前にチェックする静的解析ツール**である。

目的は以下の3つに集約される。

* 文法ミスの検出
* バグになりやすい書き方の防止
* チームの[[基礎 Part 16 Vueコード規約|コード規約]]（文化）の強制

[[ESLint]]は「正解を教えるツール」ではなく、**チームで合意したルールを機械的に守らせる装置**である。

---

## 2. [[ESLint]]の読み方と言葉の由来

* 読み方：イーエスリント
* ES：ECMAScript（JavaScriptの正式仕様）
* Lint：コードのゴミ・不純物を検出する静的解析ツール

つまり「ECMAScript用のLintツール」という意味になる。

---

## 3. [[ESLint]]がチェックする主なカテゴリ

### 3.1 文法エラー（即エラー）

```js
if (a == 1 {
```

実行すらできないコードを検出する。

---

### 3.2 バグになりやすい書き方（事故防止）

```js
if (a = 1) {
}
```

[[基礎 Part 8 宣言・代入・再代入|代入]]と比較の書き間違いなど、動いてしまうが危険なコードを検出する。

---

### 3.3 コーディングスタイル（規約）

```js
const a=1
```

```js
const a = 1
```

動作は同じだが、チームで統一しないと可読性が下がる書き方を制御する。

---

## 4. [[ESLint]]はルールエンジン

[[ESLint]]自体はルールを持たない。どのルールを適用するかは[[設定ファイル言語|設定ファイル]]で決める。

例：

```js
rules: {
  'no-var': 'error',
  'no-console': 'warn'
}
```

* error：違反するとエラー（CIで落とせる）
* warn：警告のみ
* off：無効

---

## 5. console.log が禁止されがちな理由

### 理由

* デバッグ用コードの消し忘れ防止
* 個人情報・トークンの漏洩リスク
* 本番ログのノイズ増加

### よくある設定

```js
'no-console': ['warn', { allow: ['warn', 'error'] }]
```

console.log は禁止、console.error は許可するなどの制御が可能。

---

## 6. [[ESLint]]でできる制御の種類

### 6.1 バグ防止系

* no-undef（未定義[[AI激動期における「定数」思考|変数]]）
* no-unused-vars（未使用[[AI激動期における「定数」思考|変数]]）
* eqeqeq（=== 強制）

### 6.2 設計・思想の強制

* no-var（const / let 強制）
* arrow-function 強制

### 6.3 可読性・保守性

* max-lines
* max-depth
* max-params

### 6.4 環境制御

* browser / node の区別
* window や process の誤使用防止

---

## 7. [[設定ファイル言語|設定ファイル]]を書く場所

プロジェクトのルートディレクトリに置く。

```
my-project/
├─ package.json
├─ src/
└─ .eslintrc.js
```

推奨形式は `.eslintrc.js`。
理由：コメントが書ける、条件分岐ができる。

---

## 8. 設定が多く見える理由

### 理由

* [[ESLint]]はあらゆるプロジェクトに対応する汎用ツール
* プロジェクトごとに思想が異なる
* 多くの設定はプリセットの継承

### 実際に触る場所

```js
extends: [...],
rules: {
  'no-unused-vars': 'error',
  'no-console': 'warn'
}
```

設定の大半は理解しなくても運用できる。

---

## 9. 強制力のレベル

```
コードを書く
   ↓
エディタで警告（ESLint拡張）
   ↓
コミット時にチェック
   ↓
CIでlint実行 → エラーならマージ不可
```

* レベル1：エディタでの即時警告
* レベル2：コミット時[[基礎 Part 9ブロックとブロックスコープ|ブロック]]
* レベル3：CIで[[ITと不完全さに惹かれる理由|完全]]強制

---

## 10. 正しい運用方針

* バグ系ルールは error
* 好み系は warn または [[Prettier]] に任せる
* 自動修正できるルールを増やす
* ルール追加は「事故が起きた後」に行う

[[ESLint]]は人を縛るための道具ではなく、**チームを楽にするための仕組み**として使うのが正解である。
