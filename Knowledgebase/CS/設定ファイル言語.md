# [[設定ファイル言語]]の基礎まとめ

本資料では、主要な[[設定ファイル言語]]の歴史、用途、特徴、および統一の是非について整理する。

---

## 1. 主流の[[設定ファイル言語]]と歴史

[[設定ファイル言語|設定ファイル]]形式は用途によって最適解が異なるため、多様な形式が並存している。ここでは一般的に使われる主要形式に限定し、古い順に整理する。

### [[設定ファイル言語|INI]]（1970年代〜）

最古の設定形式の一つ。Windows 3.1 などで普及し、現在も軽量な設定に利用される。階層表現が弱く、複雑な構造には向かない。

### [[設定ファイル言語|XML]]（1998〜）

タグ構造を持つ厳密な形式。Javaエコシステム、Android、Office内部構造などで広く使われた。可読性の低さや冗長さがあり、近年は減少傾向。

### [[設定ファイル言語|JSON]]（2001〜）

軽量で構造化されたデータを扱うために生まれた形式。Web API の事実上の標準。コメント不可などの制約を持つが機械処理に強い。

### [[設定ファイル言語|YAML]]（2001〜）

人間が読み書きしやすいことを重視した形式。Kubernetes、Ansible、CI/CD などインフラ領域で広く採用される。インデント依存のためミスに弱い。

### [[設定ファイル言語|TOML]]（2013〜）

[[設定ファイル言語|INI]] の簡潔さと [[設定ファイル言語|JSON]] の厳密さを併せ持つ形式として開発された。Rust と Python（pyproject.[[設定ファイル言語|toml]]）で採用され、近年利用が拡大している。

---

## 2. 現在のシェア感

用途により最適な形式が異なるため、形式ごとに強い分野が分かれている。

### [[設定ファイル言語|JSON]]

世界的に最も広く利用される。特に Web API や構造化データ処理において標準的な位置を占める。

### [[設定ファイル言語|YAML]]

インフラ構成や CI/CD など、人間が読み書きする構成ファイルで最も普及している。Kubernetes による影響が大きい。

### [[設定ファイル言語|TOML]]

最新世代として着実に普及が進んでいる。ローカル設定や開発者向け設定に採用されやすい。

### [[設定ファイル言語|XML]]

古いシステムや Android など特定領域では根強く使われ続けているが、新規採用は減少している。

### [[設定ファイル言語|INI]]

軽量な設定を中心に生き残っているが、複雑な用途には向かず徐々に役割が限定されている。

---

## 3. 統一されない理由

[[設定ファイル言語|設定ファイル]]形式は統一されていない。理由は次の通りである。

### 用途ごとに最適解が大きく異なる

人間が書く構成（[[設定ファイル言語|YAML]]）、機械が扱う構造化データ（[[設定ファイル言語|JSON]]）、堅牢でシンプルな設定（[[設定ファイル言語|TOML]]）など、目的に応じて必要な性質が異なる。

### エコシステムの歴史的背景が大きい

CloudFormation は YAML/JSON、Terraform は HCL、Rust は [[設定ファイル言語|TOML]] など、大規模エコシステムが既に定着しており、変更コストが大きい。

### フォーマットが宗教的対立になりやすい

可読性や厳密性に対する価値観が開発者ごとに異なり、一つの形式に統一することが困難である。

---

## 4. 統一の利点

統一には一定のメリットがあり、特にチーム開発では有効である。

### 学習コストの削減

設定フォーマットが統一されていると、新規メンバーが学ぶ内容が減る。

### レビューと運用の簡素化

形式が揃っているためレビューコストが減り、ドキュメントも簡潔になる。

### 内部ツールの開発効率向上

[[基礎 Part 5セミコロン・パース|パース]]処理が一種類で済むため自動化が容易になる。

---

## 5. 統一が目的化すると危険な理由

### 各形式の持つ強みを失う可能性

用途に応じた最適な形式を使えなくなることで、表現力や安全性が損なわれる。

### 既存システムの大規模な変更コスト

エコシステムを強制的に変えると、多くの利用者に負担がかかる。

### 本質から逸れる議論が生まれやすい

統一そのものが目的化すると、技術選定が合理性ではなく宗教論争に陥る。

---

## 6. 結論

[[設定ファイル言語|設定ファイル]]形式は用途、歴史、エコシステムの影響により複数の形式が併存している。統一には利点があるが、目的化すると本来の技術的合理性を損なうため、状況に応じて最適な形式を選択することが重要となる。
