# [[HTTPとRESTの関係]]

## 1. まず前提：[[HTTPとRESTの関係|HTTPとREST]]は「同列」ではない

[[HTTPとRESTの関係|HTTPとREST]]は、よく一緒に語られるが、**役割も生まれ方もまったく異なる**。

* **HTTP**：通信プロトコル（具体的なルール）
* **[[REST 概要|REST]]**：アーキテクチャスタイル（設計思想・原則の集合）

[[REST 概要|REST]]は仕様でも規格でもなく、
「どう設計すると分散システムはうまくスケールするか」を説明する**思想**である。

---

## 2. 時系列で見る：どちらが先か

### 2.1 HTTPが先に存在した

1990年代前半：

* Webを作る目的は「世界中の文書をリンクで共有すること」
* そのために生まれた要素：

  * [[URI・URL・URN|URL]]
  * HTML
  * **HTTP**

この時点では：

* [[REST 概要|REST]]という言葉は存在しない
* 「とりあえず動く仕組み」を作っていた

HTTPは 0.9 → 1.0 → [[HTTP 1.1|1.1]] と進化し、
結果的に巨大なスケールに耐えるWebが形成された。

---

### 2.2 Webはなぜか「異常にうまくスケールした」

Webの特徴：

* サーバを増やせば水平スケールできた
* キャッシュが効いた
* クライアントとサーバが疎結合だった
* 実装が自由だった

しかし当時：

* なぜこれがうまくいっているのか
* どの設計原則が効いているのか

は、理論的に説明されていなかった。

---

### 2.3 後から[[REST 概要|REST]]が「発見」された

2000年：

* Roy Fielding（[[HTTP 1.1|HTTP/1.1]]の中心人物）が博士論文で分析

やったこと：

* Webアーキテクチャを観察
* スケールしているシステムの共通点を抽出
* 設計原則として言語化

これが **[[REST 概要|REST]]（Representational State Transfer）**。

重要：

* [[REST 概要|REST]]は「設計した」のではなく「発見された」
* 成功例を**帰納的に抽象化**したもの

---

## 3. 帰納的に抽象化されたとはどういうことか

### 3.1 帰納法と演繹法の違い

* 演繹法：

  * 原理・思想が先
  * それを満たすように設計・実装

* 帰納法：

  * 実例・成功例が先
  * 共通点を抜き出して原理を作る

[[REST 概要|REST]]は **[[ITと不完全さに惹かれる理由|完全]]に後者（帰納法）**。

---

### 3.2 実際に起きていたこと

1. HTTPを使ったWebシステムが大量に作られた
2. 守っている実装、破っている実装が混在
3. それでも特にうまくスケールしたものが存在
4. それらの共通点を抜き出した

その結果、以下の制約が見えた：

* ステートレス
* 統一インターフェース
* キャッシュ
* リソース指向

これらをまとめた名前が [[REST 概要|REST]]。

---

## 4. [[HTTPとRESTの関係]]性（構造）

### 4.1 関係を図で表すと

```
[ Webの実装群 ]
   |  観察・分析
   v
[ REST（設計原則） ]

HTTPはWeb実装群の中核プロトコル
```

* HTTPは[[REST 概要|REST]]の「構成要素」ではない
* しかしHTTPは、[[REST 概要|REST]]の原則を**自然に満たしやすい**

---

### 4.2 [[REST 概要|REST]]の制約とHTTPの対応

| [[REST 概要|REST]]の制約    | HTTPでの具体例                 |
| ---------- | ------------------------- |
| ステートレス     | リクエストに必要な情報をすべて含める        |
| 統一インターフェース | GET/POST/PUT/DELETE + [[URI・URL・URN|URI]] |
| キャッシュ      | Cache-Control, ETag       |
| リソース指向     | [[URI・URL・URN|URI]]で対象を表現                 |

HTTPは[[REST 概要|REST]]を実現するために作られたわけではないが、
結果的に[[REST 概要|REST]]と非常に相性が良かった。

---

## 5. 重要な注意点

* HTTPを使っていれば[[REST 概要|REST]]になるわけではない
* JSONを返していれば[[REST 概要|REST]]ではない
* [[REST 概要|REST]]は「満たす／満たさない」が連続的

多くのAPIは：

* HTTPを使っている
* しかし[[REST 概要|REST]]の制約を一部しか満たしていない

それでも問題なく動くケースは多い。

---

## 6. 一文まとめ

HTTPが先に存在し、Webは偶然にも[[REST 概要|REST]]的な性質を持ってスケールした。
[[REST 概要|REST]]はその成功理由を、後から帰納的に抽象化した設計思想である。
